# why2025-app-repository

App distribution store for the **WHY2025 / DutchVMS badge** (ESP32-P4 + ESP32-C6,
BadgeVMS mini-OS). Public repo so the badge can fetch app metadata and binaries over
HTTPS without any embedded auth token.

Inspired by Nicolai's `app-repository` workflow (PR + CI checks + per-app metadata),
adapted for WHY's **PIE-ELF** app format (not Tanmatsu's AppFS `.bin`).

## Layout

One directory per app, named after its `unique_identifier`:

```
<uid>/
  <uid>.json      # the app manifest (WHY schema, same as apps/<app>/manifest.json)
  <binary_path>   # the pre-built PIE ELF, filename per the manifest's binary_path
index.json        # generated: the single list the badge polls
scripts/
  validate.py       # schema + ELF-magic validation (CI gate)
  generate_index.py # regenerates index.json from the app dirs
```

> Note: `cj_launcher` has `unique_identifier` `badgevms_launcher`, so its directory
> is `badgevms_launcher/` (directory name always matches the uid, validated in CI).

## Manifest schema (`<uid>.json`)

Exactly the WHY app manifest — `unique_identifier`, `name`, `author`, `version`,
`interpreter`, `metadata_file`, `binary_path`, `source`.

## index.json

Generated by `scripts/generate_index.py`. Each entry adds direct, auth-free download
URLs (`raw.githubusercontent.com/.../<uid>/<uid>.json` and `.../<uid>/<binary_path>`).
The badge fetches this one file, compares versions against installed apps, and pulls
any updated ELF.

## Submitting / updating an app

Open a PR that adds or updates `<uid>/<uid>.json` + the built ELF. CI validates the
schema and that the ELF is a real ELF. **CI does not rebuild app source** — this repo
distributes pre-built binaries only (source lives in `why2025-apps`), mirroring how
AppFS `.bin` submissions work upstream. On merge to `main`, CI regenerates `index.json`.
